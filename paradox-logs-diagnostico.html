Análisis del worker
Flujo de escritura en Paradox
1. LogDAO.registrar() guarda en RAM + JSON
↓ asíncrono, en nuevo Thread
2. ejecutarWorkerAsync() → new Thread("paradox-worker")
↓ lanza ProcessBuilder
3. ParadoxWorker.main() — nueva JVM (reset del límite HXTT)
↓ HXTT Paradox JDBC
4. asegurarTabla() + INSERT INTO LOGS
↓ stdout: "OK:id:timestamp" | "ERROR:mensaje"
5. Resultado ignorado silenciosamente por LogDAO
⚠ Si el worker falla (driver no encontrado, carpeta no accesible, límite HXTT), el log queda SOLO en JSON. La API no reporta el error.
!
Driver HXTT no en el classpath
Paradox_JDBC30.jar debe estar en lib/ y en el classpath de la JVM. Verifica que la API se lanzó con el jar.
!
Límite de 50 queries de evaluación agotado
El worker abre nueva JVM por operación precisamente para evitar esto. Si el SELECT_ALL falla, puede ser que otro proceso esté usando la misma JVM sin el workaround.
!
Carpeta paradox_data/ no existe o sin permisos
La ruta se resuelve con getProtectionDomain().getCodeSource().getLocation(). Si la estructura de carpetas cambia, la ruta puede ser incorrecta.
!
Archivo .db bloqueado por proceso anterior
Si la API crasheó con un worker en ejecución, el archivo puede quedar con lock. Reiniciar la API lo libera.
!
Thread daemon terminado antes que el worker
Si la JVM principal muere, los threads daemon se matan. El worker asíncrono puede no completarse si el servidor se para justo después del log.
